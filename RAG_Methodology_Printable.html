🛡️ Aegis Project Health Auditor: RAG Scoring Methodology

PMO-Aegis-Methodology-v1.0 • Confidential • Executive Reference

1. Purpose

The Aegis Project Health Auditor (RAG) Framework establishes a standardized, repeatable methodology to evaluate the execution health of enterprise projects. It implements a deterministic, quantitative engine that computes project health scores based on key operational signals, and then utilizes Generative AI strictly for reasoning and summary drafting after the objective score has been locked.

RAG (Red, Amber, Green) is used to prioritize PMO intervention:

  • GREEN (80.0 - 100.0): Project is healthy. No PMO intervention is required.
  • YELLOW (50.0 - 79.9): Operational slippages detected. Local PMO action required.
  • RED (0.0 - 49.9): Critical baseline breach. Immediate strategic recovery audit required.

2. Inputs

The framework maps unstructuredweekly status reports into a structured 13-field Pydantic schema:

Field Name Description
project_name Name of the target project.
planned_start_date / planned_end_date Baseline timeline boundaries.
current_progress Self-reported progress percentage or status.
budget / budget_spent Financial parameters.
milestones / delayed_milestones Delivered vs. late milestones tracker.
open_risks / blockers Active operational constraints.
stakeholder_comments Raw feedback notes.
resource_availability Headcount and staffing logs.

3. Evaluation Framework

I. Schedule (Weight: 30%)

Evaluation: Compares elapsed timeline ratio against reported progress. If actual progress is less than expected progress, a progress lag penalty is calculated:

Expected Progress = [(Current Date - Start Date) / (End Date - Start Date)] * 100
Lag = Expected Progress - Actual Progress
Schedule Score = max(0.0, 100.0 - (Lag * 1.5))

Why it matters: Identifies if work velocity is sufficient to meet deadlines.

II. Budget (Weight: 20%)

Evaluation: Measures cost overruns. If spent exceeds baseline budget, a penalty is applied based on overrun ratio:

Overrun Ratio = (Spent - Budget) / Budget
Budget Score = max(0.0, 100.0 - (Overrun Ratio * 200.0))

Why it matters: Controls cost variances and protects margins.

III. Milestones (Weight: 20%)

Evaluation: Evaluates milestone completion rates:

Milestones Score = max(0.0, 100.0 - ((Delayed Milestones Count / Total Milestones Count) * 100.0))

Why it matters: Reflects actual deliverables completed vs. planned.

IV. Risks (Weight: 15%)

Evaluation: Deducts penalty points from a 100.0 baseline for active blockers and risk severities:

  • Blocker: -25 points
  • Critical Risk: -20 points
  • High Risk: -15 points
  • Medium Risk: -10 points
  • Low Risk: -5 points

Why it matters: Quantifies active roadblocks and mitigates prospective failures.

V. Stakeholder Sentiment (Weight: 10%)

Evaluation: Audits comment logs. Scoring starts at a baseline of 80.0. A case-insensitive match on keyword lists boosts or penalizes the score (capped between 0 and 100):

  • Positive Boost (+10 points): happy, pleased, satisfied, excellent, great, good, successful, on track, confident
  • Negative Penalty (-20 points): concerned, worried, unhappy, dissatisfied, poor, issue, delay, risk, block, skeptical

Why it matters: Measures sponsor confidence and client satisfaction.

VI. Resources (Weight: 5%)

Evaluation: Scans resource logs. Starts at 100.0. If deficit keywords are detected (low, limited, deficit, shortage, missing, tight, constrained, understaffed), a deficit penalty is applied:

Resources Score = 100.0 - 50.0 = 50.0

Why it matters: Ensures adequate staffing to maintain velocity.

4. Weight Distribution

Indicator Category Weight Primary Calculation Driver
Schedule 30% Timeline Elapsed Ratio vs. Actual Progress
Budget 20% Cost Overrun Ratio
Milestones 20% Ratio of Delayed Milestones
Risks 15% Count of Blockers and Risk Severities
Sentiment 10% Keyword Scanning of Stakeholder Comments
Resources 5% Keyword Scanning of Resource Availability

5. Assumptions & Gaps Handling

When project updates are partial, Aegis applies default safety assumptions to ensure analysis continuity:

  • Dates: If start/end dates are missing, the Schedule Score defaults to 100.0.
  • Budget & Spent: If missing, the Budget Score defaults to 100.0.
  • Milestones: If empty, the Milestones Score defaults to 100.0.
  • Risks & Blockers: If empty, the Risks Score defaults to 100.0.
  • Sentiment / Resources: Sentiment defaults to 80.0. Resource status defaults to 100.0.

To preserve transparency, Aegis calculates a Data Confidence Score based on the completeness of the 13 required schema fields:

Confidence Score = (Count of non-null extracted fields / 13) * 100%

All missing parameters are explicitly logged in a Missing Data Log at the top of the report to flag gaps to the PMO (e.g. "Uncertainty: Budget spent is missing from source files..."), preventing the AI from fabricating parameters.

6. Example Evaluation

Project Alpha
  • Schedule: Start Date: 2026-05-01, End Date: 2026-10-01 (153 days). Progress reported: 35%. Evaluation Date: 2026-07-08 (Elapsed: 68 days).
  • Financials: Budget: $100,000, Spent: $110,000 (10% overrun).
  • Milestones: 3 total (2 completed, 1 delayed).
  • Risks: 1 blocker, 1 high-severity risk.
  • Sentiment: 1 positive keyword ("pleased"), 2 negative keywords ("concerned", "delay").
  • Resources: 1 deficit keyword ("limited").
Calculated Sub-Scores:
  1. Schedule Score: Expected = 44.4%, Lag = 9.4%. Score = 100.0 - (9.4 * 1.5) = 85.9
  2. Budget Score: Overrun = 10%. Score = 100.0 - (0.10 * 200.0) = 80.0
  3. Milestones Score: Delay = 33.3%. Score = 100.0 - 33.3 = 66.7
  4. Risks Score: Blocker (25) + High Risk (15) = 40. Score = 100.0 - 40 = 60.0
  5. Sentiment Score: Base = 80.0 + 10.0 - 20.0 - 20.0 = 50.0
  6. Resources Score: Deficit penalty = 50. Score = 100.0 - 50 = 50.0
Weighted Aggregation:
Overall Score = (85.9 * 0.30) + (80.0 * 0.20) + (66.7 * 0.20) + (60.0 * 0.15) + (50.0 * 0.10) + (50.0 * 0.05)
Overall Score = 25.77 + 16.00 + 13.34 + 9.00 + 5.00 + 2.50 = 71.6

Weighted Health Score: 71.6 / 100 → YELLOW status.
